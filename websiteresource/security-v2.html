Security

The record is only worth keeping if it's still intact.

Here is how we protect it, restrict access to it, handle the parts we get wrong, and how to tell us when you find something we missed.

Our posture, up front

Designed to recognized security principles. Not yet independently certified.

The practices on this page reflect how the system was built from the beginning — least privilege, defense in depth, cryptographic record integrity, and minimal data retention — and they map to SOC 2 Type II and ISO 27001 principles.

We do not currently hold an independent audit certification. Operators with a formal vendor-review process should treat this page as a statement of practice. Contact security@boatcheckin.com for our current security questionnaire.

Principles we build to

Six decisions made on day one.

Structural commitments that shape every feature we ship.

I

Least privilege, by default.

Every operator sees only their own data. Every guest sees only their trip. Database-level rules enforce these boundaries.

II

Record integrity is non-optional.

Waivers and attestations are SHA-256 hashed at signing. Any modification is detectable. Audit logs are sealed at trip completion.

III

Encryption everywhere.

TLS 1.3 in transit. AES-256-class at rest. No cleartext storage of credentials, tokens, or guest PII, ever.

IV

Minimize what we collect.

No silent telemetry on guests. No behavioral profiling. No lookalike-audience exports. What isn't collected can't be exposed.

V

Every action is attributable.

State changes produce an audit trail with actor, timestamp, method, and client metadata.

VI

No security through obscurity.

Public surfaces are rate-limited, authenticated, validated, and logged — as if every request were adversarial.

Protecting the record

How the trip record is kept intact.

What stops a waiver from being altered, a manifest from being rewritten, or an attestation from being manufactured after the fact.

Transport

Encrypted in transit, modern cipher suites only.

All traffic protected by TLS 1.3. Older versions disabled. HSTS enforced. No unencrypted channel has ever been used.

Storage

Encrypted at rest, minimal privilege inside the database.

Enterprise-grade relational database with AES-256 encryption. Row-level access rules isolate operator and guest data. Humans do not hold direct production credentials.

Integrity

Signatures and attestations are cryptographically hashed.

SHA-256 hash stored alongside each record. If altered even by a single character, re-hashing produces a different digest. Tampering becomes visible.

Audit trail

State changes are logged with actor, time, and method.

Audit entries are append-only — no API or UI edits or deletes them. After trip completion, the log is sealed.

Backups

Continuous backups with tested restores.

Point-in-time recovery across a rolling retention window. We run restore drills on a scheduled cadence — a backup that hasn't been tested is an assumption.

Restricting access

Who can touch the record — and how.

Access control sits at multiple layers, so no single mistake exposes an account.

Operators

Short-lived, scoped sessions.

Strong-password or passwordless authentication. Sessions are short-lived and bound to the originating client. A session for one operator cannot read another operator's data.

Guests

Signed, scoped, time-limited trip links.

Cryptographically signed tokens bind links to exactly one trip. Guest sessions are disposable after boarding completion.

Captains

Versioned, revocable, expiring snapshot tokens.

No login required — but the HMAC-signed token has an explicit expiry and version counter. Re-issuing a snapshot invalidates the prior link.

Internal

Production access only when necessary, only when logged.

No routine access to operator or guest data. Scoped, auditable tooling. Secrets in a dedicated store, rotated on schedule, never in source code.

Public surface

Every endpoint is rate-limited and validated.

Bot challenges where appropriate. Schema-validated on the server. Suspicious patterns produce on-call alerts.

Vulnerability Disclosure

Find something we missed? Tell us.

We welcome responsible-disclosure reports from security researchers, operators, and curious engineers.

In scope

What we want to hear about

Anything that would let an unauthorized party read, modify, or destroy operator or guest data.

  • Authentication and session flaws
  • Authorization bypass across operators
  • Data exposure via direct object reference
  • Injection vulnerabilities
  • Cross-site scripting
  • Token generation weaknesses
  • Audit-log integrity abuse
Out of scope

Lower priority reports

Still welcome, but typically not treated with the same urgency:

  • Missing headers without a concrete exploit
  • Self-XSS without weaponization
  • Rate-limiting on marketing pages
  • Clickjacking on unauthenticated pages
  • Automated scanner reports without PoC
  • Volumetric denial of service
Safe harbor

Good-faith research is welcome — within these lines.

We will not pursue legal action against researchers who act in good faith, report promptly, do not exfiltrate unnecessary data, and allow us reasonable remediation time.

Please do not test against real operator accounts. Please do not run load-generating automated scans. We don't run a paid bounty but acknowledge valid findings publicly.

If something goes wrong

Incident communication, before we need it.

If a security incident affects operator or guest data, affected operators receive direct email notice without waiting for a public announcement. We support operators with the technical details they need for downstream notifications.

For service-availability incidents, we publish updates to a status page so operators don't need to file a ticket to learn if the issue is on our side.

In every case, we prefer direct, specific, and early communication over polished marketing language after the fact.
